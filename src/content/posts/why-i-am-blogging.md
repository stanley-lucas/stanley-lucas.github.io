I've been thinking about this for a while.

Not the blogging part — the *why* part.

I'm a Computer Engineer and an Air Traffic Controller based in Curitiba, Brazil. I operate safety-critical systems for a living. I also build software — and I have a long-term goal that most people would call unrealistic: contributing to aerospace software engineering at the highest level possible.

So why blog?

## This is my memory

I learn a lot. I forget a lot. I've had conversations, built things, read things — and six months later I can't reconstruct the reasoning behind a decision I made. This blog is my external memory. A place I can come back to and read what I was thinking, what I built, and why.

If it helps someone else along the way, great. But honestly, I'm writing this for future me.

## This is my build log

I'm currently starting a project — a Falcon 9 launch telemetry dashboard built with Python, PostgreSQL, and Streamlit, using public SpaceX data. Every decision I make, every problem I hit, every concept I have to learn to move forward — I'll document it here.

Not polished tutorials. Raw build logs.

## The intersection I live in

ATC taught me system design before I knew the term.

Managing multiple aircraft simultaneously, interpreting radar telemetry in real time, operating systems where a wrong call has irreversible consequences — that's where I learned what fault tolerance, latency, and observability actually mean in practice.

Now I'm building software in the same domain. Space systems, launch data, orbital mechanics. The stakes feel familiar. The tools are new.

## What to expect here

- Project build logs (starting with the Falcon 9 telemetry dashboard)
- Concepts I'm learning: system design, data engineering, ML for anomaly detection
- Reflections on operating at the intersection of two demanding fields
- Honest writing — including the parts where I don't know what I'm doing

This is not a tutorial blog. It's a thinking blog.

Let's build.
